## 1. プロジェクト概要

### 1.1 目的（※ゴール変更反映後）

本プロジェクトは、瑞穂町コミュニティバスを対象とした  
**バスロケーションシステムの技術実証（PoC）**である。

従来の目的であった「住民向け完成UIの提供」ではなく、  
本PoCの主目的は以下に置く。

> **1台のバス車両において、清瀬市で採用されている  
>「GPS取得 → GTFS-Realtime生成 → 公開」モデルと同等の技術要件を、  
>Google品質基準を満たす形で、いくらの費用で実現できるかを実証すること**

これにより、
- 将来的に清瀬モデル相当の本格導入を行う場合の**比較材料**
- 「1台構成」「複数台構成」それぞれの**費用感・スケール性**

を示す **行政向け技術・費用実証レポート**の作成をゴールとする。

## 2. 対象範囲・前提条件

### 2.1 対象範囲（Scope）

* バス車両：**1台**
* Module A（車載・GPS送信）
* Module B（受信・ログ保存・GTFS-RT生成・公開）

### 2.2 非対象（Out of Scope）

* 住民向け完成UIの提供
* 複数台での本格運用
* 運転士による操作を前提とした機器・UI

---

## 3. 共通要件

### 3.1 更新頻度要件

* **通常運用：10秒間隔程度**
* **通信劣化時でも最大30秒以内**
* 30秒を超える欠損は品質劣化としてログに記録する

※ Google Maps 等での採用基準（30秒以内更新）を満たす設計とする。

---

### 3.2 運用要件

* 運転士による操作を必要としない完全自動運用
* 視界・運転操作の妨げにならない設置
* 車両電源ON/OFFに追従して自動起動・停止

---

### 3.3 電源要件

* 車両電源：24V
* ヒューズボックスから取得
* DC/DCコンバータにより機器電圧へ変換

---

## 4. Module A：車載デバイス要件

### 4.1 概要

バス車両に搭載し、GPSによる位置情報を取得し、
Module Bへ定期的に送信する。

### 4.2 機能要件

| ID   | 項目     | 要件                                | 優先度 |
| :--- | :----- | :-------------------------------- | :-- |
| A-01 | ハードウェア | 汎用デバイス（スマートフォン／IoTモジュール等）を使用可能とする | 高   |
| A-02 | 計測データ  | 緯度・経度・タイムスタンプ・精度は必須。速度・方位は任意      | 高   |
| A-03 | 送信頻度   | 約10秒間隔でHTTPS送信                    | 高   |
| A-04 | 自動化    | 車両給電開始で自動起動、停止で終了                 | 高   |
| A-05 | 冗長性    | 通信断時は再接続を自動試行                     | 中   |

### 4.3 非機能要件

* 特定メーカー・特定端末に依存しない
* 再現性・調達容易性を重視

---

## 5. Module B：サーバ要件

### 5.1 概要

Module A から送信された位置情報を受信し、
GTFS-JPと照合したうえで GTFS-Realtime を生成・公開する。

### 5.2 機能要件

| ID   | 項目        | 要件                              | 優先度 |
| :--- | :-------- | :------------------------------ | :-- |
| B-01 | データ受信     | HTTPSエンドポイントでJSON受信             | 高   |
| B-02 | ログ保存      | 生GPSデータを時系列で保存                  | 高   |
| B-03 | GTFS照合    | GTFS-JPに基づき運行状態を判定              | 高   |
| B-04 | GTFS-RT生成 | Protocol Buffers形式で feed.pb を生成 | 高   |
| B-05 | 公開        | 外部から取得可能なURLで公開                 | 高   |
| B-06 | ログ蓄積      | 生成したGTFS-RT内容を保存                | 中   |

※ 初期段階では **VehiclePosition** を中心とし、TripUpdate（遅延）は段階的に導入可能とする。

---

## 6. インタフェース仕様（Module A → B）

### 6.1 通信方式

* HTTPS POST
* JSON形式

### 6.2 送信データ例

```json
{
  "vehicle_id": "bus_001",
  "timestamp": "2026-02-15T10:23:10+09:00",
  "lat": 35.771234,
  "lon": 139.345678,
  "accuracy": 12.5,
  "speed": 30.5,
  "heading": 120.0
}
```

### 6.3 欠損時の扱い

* 欠損があってもHTTP 400は返さない
* 保存・処理できない場合はスキップしログのみ残す

---

## 7. ログ保存要件（重要）

### 7.1 保存対象

1. **生GPSログ**

   * vehicle_id
   * timestamp
   * lat / lon
   * accuracy
   * 受信結果
2. **生成GTFS-Realtimeログ**

   * feed内容
   * 生成時刻

### 7.2 保存目的

* 実証実験終了後に以下を算出するため

  * 平均更新間隔
  * 最大更新間隔
  * 30秒超欠損率
  * 稼働率（日別）
  * 品質と費用の関係

---

## 8. 品質評価指標

* 平均更新間隔（秒）
* 最大更新間隔（秒）
* 30秒超更新欠損の発生割合
* feed timestamp の鮮度
* 実稼働日数／実証期間

---

## 9. スケール検討（参考）

本PoCは1台構成とするが、

* 複数台展開時の通信費増加
* サーバ処理負荷
* 運用・保守コスト

については、実証ログをもとに将来検討を行う。
