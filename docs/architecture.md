# システムアーキテクチャ（PoC） — 瑞穂町コミュニティバス バスロケーション

## 1. 目的と設計方針

本PoCは、瑞穂町コミュニティバス（車両1台）を対象に、以下を満たす「最小構成のバスロケーション基盤」を構築する。

- 車載GPSから位置情報を取得し、約10秒間隔で更新
- Google GTFS-Realtime（以下 GTFS-RT）仕様に準拠した feed を生成
- Google Maps 等への提供モデル（「GPS → GTFS-RT → 配信」）を成立させる
- 実証期間中のログ蓄積と、事後の集計・費用実証レポート作成を可能にする

設計上の最重要方針は次の通り。

- **運転士操作不要**：車載機は完全自動で動作
- **再現性**：特定ベンダ製品に依存しない（部材選定・方式の自由度を確保）
- **セキュリティ境界の明確化**：車載端末を信頼せず、サーバ側で検証・整形してからDBへ保存
- **将来拡張**：便（trip）推定、遅延判定、複数台展開に接続できる構造

---

## 2. 全体構成（モジュール分割）

本PoCは役割に応じて以下のモジュールで構成する。

- **Module A（車載・エッジ）**  
  GPS測位し、約10秒間隔で現在地をサーバへ HTTPS POST 送信する。
- **Module B（サーバ）**  
  Module A からの位置情報を受信し、ログ保存・整形・GTFS-RT生成・公開を行う。
- **Module C（任意）**  
  デモ・検証のための簡易フロントエンド（PoC必須ではない）。

---

## 3. 論理アーキテクチャ（データフロー）

### 3.1 データフロー概要

```mermaid
flowchart LR
  A[Module A: 車載GPS端末] -->|HTTPS POST /gps| B[Module B: Cloud Function(Gen2)]
  B -->|write| D[(Firestore)]
  B -->|GET /gtfs_rt| C[利用者/検証者]
  C -->|HTTP GET feed.pb| B
  B -->|read latest| D
  B --> E[GTFS-RT feed.pb]
````

* Module A は **Firestoreへ直接書き込まず**、必ず Module B の受信APIを経由する。
* Module B が受信データを整形し、Firestoreに「生ログ」と「最新状態」を保存する。
* Module B は Firestore の最新状態から GTFS-RT（VehiclePosition）を生成し、外部提供可能なURLで返す。

### 3.2 APIエンドポイント（Module B）

* **POST /gps**
  車載端末からGPSデータを受信する入口。認証・検証・保存を行う。
* **GET /gtfs_rt**
  GTFS-RT（Protocol Buffers）を生成して返す提供口。外部システム（Google側確認、検証者）が取得可能。

## 4. 物理アーキテクチャ（クラウド実装）

### 4.1 採用コンポーネント

* Firebase / Google Cloud 上で以下を利用する。

  * **Cloud Functions (Gen2)**：HTTPS API（受信・提供）
  * **Cloud Run（Functions Gen2の実行基盤）**
  * **Firestore**：GPSログ・最新状態の保存
  * （必要に応じて）Cloud Logging：運用監視・トラブルシュート

### 4.2 リージョン

* PoC初期段階では **関数（Functions Gen2）のデフォルトリージョン**で構築されている。
* 将来的には、Firestore 等のリージョンと整合させ、遅延・運用性・コストを見ながら **リージョン統一**を検討する。

## 5. データ設計（Firestore）

### 5.1 目的

PoCでは「後から検証できる」ことが重要であるため、次の2種類を保存する。

* **生ログ（監査・検証・品質評価用）**：欠損・異常値・通信状況を含めて記録
* **最新状態（提供用）**：GTFS-RT生成に必要な最新の採用データのみ保持

### 5.2 保存構造（例）

* `gps_logs`（コレクション）

  * 受信したGPSの生データ（必要に応じて accepted / reject_reason 等を付与）
* `vehicles/{vehicle_id}/state/latest`

  * 車両ごとの最新位置（採用済みデータ）

> 注：実際のフィールド定義・例は `docs/requirements_v1.1.md` および `docs/api-spec.md` に記載する。

## 6. セキュリティ設計（なぜDB直書きにしないか）

### 6.1 結論

**Module A からFirestoreへ直接書き込みを行わず、Module B（受信API）を経由する設計**とする。

### 6.2 理由（行政向けの説明ポイント）

* **車載端末は盗難・改造・複製のリスクがある**
  端末にDB書き込み権限（資格情報）を持たせると、漏洩時に不正書き込み・大量書き込みによる課金増・改ざんが起こり得る。
* **サーバ側で検証・整形・拒否が可能**
  受信時に APIキー検証、異常値判定、頻度・形式の検査を行い、DBを「信頼できるデータ」に保つ。
* **運用・拡張が容易**
  後段の「便（trip）推定」「遅延判定」「欠損補間」など、サーバ側ロジックの追加・更新が容易。

### 6.3 認証方式

* Module B（受信API）は **APIキー**によりアクセス制御する。
* 車載端末はリクエストヘッダ `X-API-KEY` を付与し、サーバ側で一致を検証する。
* APIキーは Cloud Run / Functions の **環境変数**で管理し、リポジトリには含めない。

## 7. 可用性・運用（PoCで見るべき指標）

### 7.1 更新頻度・欠損

* 通常：10秒程度
* 通信劣化時でも最大30秒以内を目標
* 欠損時は「次回更新で復帰」できること（再送キューの有無はPoC要件に応じて判断）

### 7.2 監視・トラブルシュート（最低限）

* 受信API（/gps）のステータスコード（401/4xx/5xx）
* 受信ログ件数（車両単位、時間単位）
* Firestore書き込み頻度（過剰増加の検知）
* GTFS-RT取得成功率（/gtfs_rt の200率）

## 8. リポジトリ構造（実装の位置づけ）

本リポジトリは以下の構造を持つ。

```
.
├─ backend/      # Module B（Firebase Cloud Functions / Python）
├─ frontend/     # Module C（デモ・検証用）
├─ edge/         # Module A 関連（車載側の実装・メモ・スクリプト等）
├─ docs/         # 要件定義・設計・手順・報告書ドキュメント
├─ data/         # 参考データ、加工データ（運用方針により整理）
├─ functions/    # 将来拡張/補助関数等（運用方針により整理）
└─ references/   # 参照資料、外部リンク、検証メモ等
```

> 注意：`backend/venv/` はローカル開発用の仮想環境であり、通常はGit管理対象外とする（`.gitignore` で除外する）。

* **リージョン最適化**：Firestore/Functions のリージョン統一、レイテンシと運用コストを最適化
